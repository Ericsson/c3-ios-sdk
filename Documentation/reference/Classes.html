  • C3AccountData

    Represents account data that is persistent and unique to each client. It is synchronized between all clients of the same user, and can be used for e.g. settings or unread notifications.

    Account data can be scoped either to the user or to a room. It is accessed through C3Client.accountData and C3Room.accountData.

    Each account data is a key-value pair store that emits events when values are changed.

    Version

    1.0.0

    Events

    • update: Emitted when any account data entry changes. Receives C3DataUpdate instance.
    • update:<key>: Emitted when the value associated with the given key is changed. Receives the new value, if any.

  • C3Component

    A base class for components that can be attached to Calls.

    Subclasses have a number of lifecycle methods that can be overriden to implement actions during various parts of a call.

    Components that are attached will be able to interact with other remote components, as long as the components are compatible. An example is two users in a call that both attach a data share component to the same point. The two data shares will then synchronize data, as long as the calls are connected.

    The purpose of the C3Component.label and C3Component.remoteLabel configuration is to make sure that components are compatible before connecting. A component is compatible with another component if the label is equal to the remote label, or match in the case where the remote label is a regular expression. If components are not compatible they will not be connected, so the peer will not show up in the C3AttachPoint.peers collection.

  • C3MediaNode

    A base class that all pure media nodes inherit from.

    Classes inheriting from media node are plain nodes with input and output ports directly attached.

    A media node has zero or more input and output ports, which can be connected to other nodes. Media nodes are always connected from one output port to one input port, using the connect method of either the output port or the media node.

    When using the connect method of a media node, the output port to be used is resolved by recursively looking up the output property of the media node. The same method is used when a media node is used as the target for a connection, but in that case an input port is looked up via the input property.

    Because the input and output property lookups are recursive, it is easy to create simple container nodes that have a number of internal media nodes, and then expose input and output ports as properties.

  • C3PowerLevelsEdit

    A class which provides a fluent API for modifying a power levels object. It cannot be instantiated directly and can only be created through the static factory methods as well as C3Room.editPowerLevels().

    Many methods are verbs, but they simply set the power level required to execute the matching action, with the exception of C3PowerLevelsEdit.commit.

    No actual action is taken until the C3PowerLevelsEdit.commit method is called. The commit method is only available if the instance was obtained from a C3Room.editPowerLevels() call. It is also possible to create an instance with one of the static factory methods and then pass the instance to C3Room.setPowerLevels(_:success:failure:).

    Version

    1.0.0

    Example

    room.editPowerLevels()
    .kick(has: 60)
    .addState(has: 0)
    .user(promotedUser, has: 60)
    .event(my.customer.event, has: 20)
    .commit({
        print("Power levels updated!)
})
    


    room.setPowerLevels(
    C3PowerLevelsEdit.fromDefault(client.user!),
    success: { room in
        print(Power levels reset)
})

  • C3Room

    A representation of a room that users join in order to communicate.

    Version

    1.0.0

    Events

    • event:<type>: Same as event, but listens to a specific event type. Receives C3Event instance.
    • alias: Emitted when the alias of the room changes.
    • avatar: Emitted when the avatar of the room is changed. Receives C3ImageResource instance or nil.
    • call: Emitted when there is an incoming call to this room. Incoming calls are stopped and need to be started with C3Call.start() before a connection is set up. Receives C3Call instance.
    • event: Emitted when a new event is triggered in the room. Will not trigger when loading old events. Receives C3Event instance.
    • events: Emitted when the list of events in the room change. This event does not have any parameters, the updated list of events can instead be accessed via the C3Room.events property.
    • guestAccessRule: Emitted when the guest access rule of the room changes. Receives C3GuestAccessRule instance.
    • historyVisibility: Emitted when the history visibility of the room changes. Receives C3HistoryVisibility instance.
    • invite: Emitted when a user is invited to the room. Receives the user that was invited.
    • invited: Emitted when the invited user collection is changed. This event does not have any parameters, the updated list of users can instead be accessed via the C3Room.invited.
    • join: Emitted when a user joins the room. Receives the user that joined.
    • joinRule: Emitted when the join rule of the room changes. Receives C3JoinRule instance.
    • leave: Emitted when a user leaves the room. Receives the user.
    • members: Emitted when the members user collection is changed. This event does not have any parameters, the updated list of users can instead be accessed via the C3Room.members property.
    • membership: Emitted when the authenticated users membership of the room is changed. Receives C3RoomMembership instance.
    • name: Emitted when the name of the room is changed. Receives the new name.
    • C3Room.otherMembers: Emitted when the C3Room.otherMembers user collection is changed. This event does not have any parameters, the updated list of users can instead be accessed via the C3Room.otherMembers property.
    • powerLevels: Emitted when the power levels of the room change. Receives C3PowerLevelsReader instance.
    • state: Emitted when a state of the room is changed. Receives C3RoomState instance.
    • topic: Emitted when the topic of the room is changed. Receives the new topic.
    • type: Emitted when the type of the room is changed.
    • typing: Emitted when the list of users who are currently typing is changed. This event does not have any parameters, the updated list of users can instead be accessed via the C3Room.typing property.
    • state:<stateType>: Same as C3Room.state, but listens to a specific state type.
    See more

  • C3RoomState

    Represents a state in a room identified by a state type, and synchronized between all clients.

    Each room state is a key-value pair store that emits events when values are changed.

    The room state also has a default value which can be accessed by omitting the key parameter to any of the methods. The default value is handy for simple states such as setting a hex color for the room.
